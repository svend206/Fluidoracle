Component Performance Verification Protocol

A structured framework for capturing real-world industrial component performance data after installation — closing the loop between recommendations and outcomes.

The problem

Engineers select industrial components based on manufacturer specifications, application experience, and peer recommendations. When a component fails — or performs better than expected — that information rarely makes it back to the people who recommended it, the engineers who will face the same decision next year, or the manufacturers who need to understand real-world performance gaps.

The component selection process is one-directional. Recommendations flow from specifications to engineers. Outcomes flow nowhere.

CPVP closes this loop. It defines when to follow up, what to ask, how to capture the data, and what to do with it — in a format that works across any component category and any platform that chooses to implement it.

What is CPVP

The Component Performance Verification Protocol is a structured post-installation follow-up system. It defines three check-in interactions at defined intervals after a component is installed, each capturing progressively deeper performance data.

CPVP is designed to be implemented by any platform, organization, or engineering team that makes component recommendations and wants to systematically track outcomes. It is not FluidOracle-specific — it is an open standard published under CC BY 4.0.

Design principles

Engineer-controlled timing The commissioning clock starts when the engineer confirms installation — not when the recommendation was made.
Conversation, not forms Check-ins are conversations that adapt to what actually happened, not fixed fields to fill in.
Value during check-in Each interaction delivers something useful to the engineer — interpretation, projections, advice — not just data extraction.
Partial data is valid Incomplete check-ins are captured and stored. Partial data is better than no data.
Append-only records Check-in records are never overwritten. Corrections add dated notes rather than replacing prior data.
Trust by design Every email references something only the platform would know. The engineer should recognize the follow-up immediately.

Three-stage structure

CPVP consists of three follow-up interactions. Timing is measured from installation confirmation, not from the original consultation date.

Before the three stages, a lightweight pre-commissioning contact (48–72 hours after the initial consultation) asks whether the engineer moved forward. This is not a data collection event — it is the natural moment to confirm installation intent and start the commissioning clock. If the engineer confirms installation, Stage 1 follows 24 hours later. If there is no response after a 2-week follow-up, the contact ceases.

Stage 1 — Installation record

1
Installation Record
24 hours after installation confirmation

Captures installation conditions while they are fresh. The primary purpose is to establish a verified baseline — what was actually installed, under what conditions, with what initial readings.

  • Confirmation of component actually installed (may differ from recommendation)
  • Initial performance readings at operating conditions (e.g., baseline differential pressure)
  • Condition of replaced component (wear patterns, failure evidence)
  • Any deviations from recommended configuration
  • Baseline fluid or system sample, if taken

Stage 2 — Performance verification

2
Performance Verification
4–6 weeks after installation confirmation

Captures early performance data after sufficient operating time to observe real behavior. Performance readings are compared against the Stage 1 baseline to calculate trends and project forward.

  • Current performance readings compared to Stage 1 baseline
  • Trend and projection (e.g., loading rate, projected service life)
  • Follow-up sample results compared to target specification from original consultation
  • Status of original problem (resolved, improved, unchanged)
  • Any anomalies or unexpected behavior

Stage 3 — Service life record

3
Service Life Record
6–12 months after installation confirmation

The highest-value data collection event. Captures long-term reliability, failure modes, and the engineer's holistic judgment on recommendation quality.

  • Component status: still in service, or replaced (when, why)
  • Total service life if replaced
  • Component condition at end of life
  • Failure mode (from vertical-specific taxonomy)
  • Original problem resolution over the full period
  • Cost impact, if quantifiable
The key question — asked at Stage 3

"Knowing what you know now, would you select the same component for this application?"

This is the single most valuable data point CPVP captures. It is a holistic judgment that incorporates everything the engineer experienced: installation difficulty, early performance, reliability, cost versus value, manufacturer support. It cannot be inferred from specifications. It can only come from an engineer who lived with the component.

Email requirements

Follow-up emails are the primary interface for most engineers. Every design decision must prioritize deliverability and immediate recognition.

  • Subject lines must be specific. Reference the component, application, and system. Not "How's your filter working?" but "Your Pall HC9600 in stamping press HPU — installation check-in." The engineer should recognize it before opening.
  • Email body includes a consultation summary. Application, component installed, date of consultation, key parameters. Enough to confirm legitimacy without requiring a click.
  • Tokenized links are time-limited, not single-use. 7-day expiry with a re-authentication fallback. Engineers get interrupted.
  • Plain-text alternative required. Engineers at industrial facilities often have aggressive email filtering.
  • Unsubscribe in every email. Required by law and a trust signal.
  • Consistent sender identity. Sender domain, link domain, and platform name must match. Domain changes reset trust with both email providers and engineers.
  • DKIM, SPF, and DMARC authentication required. Non-negotiable for industrial email deliverability.

The PDF commissioning report (see Data Model below) is attached to each Stage 1, 2, and 3 email. Engineers can file the PDF in their maintenance system regardless of whether they click through. Maximum value delivery with minimum friction.

Data model

CPVP defines the following minimum data model. Implementing platforms may extend this with vertical-specific fields.

Consultation record

  • consultation_id — unique identifier
  • operating_context — system-level data (fluid, pressure, flow, temperature, application). Shared across verticals for the same physical system.
  • recommendation — component recommended, with reasoning and cited sources
  • consultation_date

Commissioning record (per stage)

  • stage — installation | performance | service_life
  • stage_date
  • component_installed — confirmed at Stage 1; may differ from recommendation
  • implementation_status — implemented | modified | not_implemented | pending
  • performance_readings — vertical-specific measurements (extensible)
  • baseline_reading — Stage 1 reference value for trend calculation
  • problem_resolution_status — resolved | improved | unchanged | worsened
  • failure_occurred — boolean
  • failure_mode — from vertical-specific taxonomy
  • would_select_again — yes | no | uncertain (Stage 3 only)
  • additional_notes — free text
  • corrections — append-only list of dated corrections to prior stage data

Commissioning report (PDF)

A single PDF document that grows with each check-in. Sections are added at each stage; prior sections are never overwritten. Corrections appear as dated addenda. The report is generated at the close of each check-in conversation and attached to subsequent check-in emails.

Implementing CPVP

CPVP is designed to work across any component category. Implementing platforms register vertical-specific configuration — check-in parameters, performance metrics, failure mode taxonomies — and the shared protocol infrastructure handles scheduling, email delivery, conversation flow, and PDF generation.

The minimum requirements for a compliant CPVP implementation:

  • Three check-in stages at the specified intervals, triggered by installation confirmation
  • Append-only records with dated corrections
  • Email trust requirements met (specificity, authentication, unsubscribe)
  • The key Stage 3 question asked and recorded
  • A PDF commissioning report delivered to the engineer at each stage
